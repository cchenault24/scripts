# Preserving Custom Configurations

This guide explains how to keep your custom shell configurations when using zsh-setup.

## The Problem

When you run `zsh-setup install`, it regenerates your `.zshrc` file. This ensures a clean, consistent configuration but can overwrite your custom aliases, functions, and settings.

## The Solution: `.zshrc.local`

zsh-setup now supports a `.zshrc.local` file for your custom configurations. This file:
- ✅ Is never overwritten by zsh-setup
- ✅ Is automatically sourced by the generated `.zshrc`
- ✅ Survives multiple installations and updates
- ✅ Keeps your customizations separate from generated config

## Usage Patterns

### Option 1: Automatic Migration (Recommended)

If you already have a zsh-setup generated `.zshrc` with custom configurations:

```bash
# Just run install - custom configs are automatically extracted
./zsh-setup install
```

The install process will:
1. Detect custom configurations in your current `.zshrc`
2. Extract them to `~/.zshrc.local`
3. Backup your old `.zshrc` to `~/.zsh_backup/`
4. Generate a new `.zshrc` that sources `.zshrc.local`

### Option 2: Manual Migration

If you want more control over the migration:

```bash
# Run the migration script first
./migrate-custom-configs.sh

# Then run install
./zsh-setup install
```

The migration script will:
- Show you what will be migrated
- Let you review before confirming
- Create `.zshrc.local` with your custom configs

### Option 3: Manual Creation

If you prefer to manually manage your configs:

```bash
# Create .zshrc.local
touch ~/.zshrc.local

# Add your custom configurations
cat >> ~/.zshrc.local <<'EOF'
# My custom aliases
alias ll='ls -lah'
alias gs='git status'

# My custom functions
function mkcd() {
    mkdir -p "$1" && cd "$1"
}

# Custom environment variables
export MY_VAR="value"
EOF

# Run install
./zsh-setup install
```

## What Gets Migrated?

### Automatically Migrated ✅
- Custom aliases
- Custom functions
- Custom environment variables
- Plugin-specific configurations you added
- PATH modifications you made

### Not Migrated (by design) ⚠️
- Comment-only lines
- The `source ~/.zshrc.local` line itself
- Empty lines
- The section header comments

### Not Supported ❌
- If your `.zshrc` is **not** generated by zsh-setup, automatic migration won't work
- In this case, manually create `.zshrc.local` with your custom configs

## File Structure

After installation, your setup will look like:

```
~/.zshrc              # Generated by zsh-setup
├── Oh My Zsh configuration
├── Plugin settings
├── Environment setup
└── sources ~/.zshrc.local

~/.zshrc.local        # Your custom configurations
├── Your aliases
├── Your functions
└── Your custom settings

~/.zsh_backup/        # Timestamped backups
└── .zshrc_20240102_153045
```

## Examples

### Example 1: Aliases

**.zshrc.local:**
```bash
# Git shortcuts
alias gs='git status'
alias ga='git add'
alias gc='git commit'
alias gp='git push'

# Directory shortcuts
alias proj='cd ~/projects'
alias docs='cd ~/Documents'
```

### Example 2: Functions

**.zshrc.local:**
```bash
# Create directory and cd into it
function mkcd() {
    mkdir -p "$1" && cd "$1"
}

# Extract any archive
function extract() {
    if [ -f "$1" ]; then
        case "$1" in
            *.tar.bz2)   tar xjf "$1"     ;;
            *.tar.gz)    tar xzf "$1"     ;;
            *.zip)       unzip "$1"       ;;
            *)           echo "'$1' cannot be extracted" ;;
        esac
    fi
}
```

### Example 3: Environment Variables

**.zshrc.local:**
```bash
# Development environment
export EDITOR="code"
export VISUAL="code"

# Project-specific
export PROJECT_ROOT="$HOME/projects"
export DATA_DIR="$HOME/data"

# API Keys (use environment-specific files in production)
export DEV_API_KEY="dev_key_here"
```

### Example 4: Conditional Loading

**.zshrc.local:**
```bash
# Work-specific configs (only load at work)
if [[ "$(hostname)" == *"work"* ]]; then
    export WORK_VAR="value"
    alias vpn='sudo openconnect vpn.company.com'
fi

# Personal machine configs
if [[ "$(hostname)" == "my-macbook" ]]; then
    export PERSONAL_VAR="value"
fi
```

## Troubleshooting

### "My custom configs aren't being loaded"

Check if `.zshrc.local` exists and is being sourced:
```bash
# Check if file exists
ls -la ~/.zshrc.local

# Check if .zshrc sources it
grep "zshrc.local" ~/.zshrc

# Manually source to test
source ~/.zshrc.local
```

### "Migration didn't extract everything"

The automatic migration only extracts content from the "User Custom Configurations" section. If you have custom configs elsewhere in your `.zshrc`, you'll need to manually move them:

```bash
# View your current .zshrc
cat ~/.zshrc

# Manually add missing configs to .zshrc.local
nano ~/.zshrc.local
```

### "I want to start over"

```bash
# Remove .zshrc.local
rm ~/.zshrc.local

# Restore from backup if needed
ls ~/.zsh_backup/
cp ~/.zsh_backup/.zshrc_20240102_153045 ~/.zshrc

# Or run fresh install
./zsh-setup install --no-backup
```

## Best Practices

1. **Keep it simple**: Only put custom configurations in `.zshrc.local`
2. **Comment your code**: Future you will thank present you
3. **Group by category**: Organize aliases, functions, and env vars separately
4. **Version control**: Consider adding `.zshrc.local` to your dotfiles repo
5. **Test changes**: Source `.zshrc.local` after edits to catch syntax errors

## Advanced: Multiple Local Files

You can split configurations into multiple files:

**.zshrc.local:**
```bash
# Source additional config files
[[ -f ~/.zshrc.aliases ]] && source ~/.zshrc.aliases
[[ -f ~/.zshrc.functions ]] && source ~/.zshrc.functions
[[ -f ~/.zshrc.work ]] && source ~/.zshrc.work
```

This keeps your customizations organized and modular.

## Migration Safety

The migration process is safe:
- ✅ Original `.zshrc` is backed up before any changes
- ✅ Existing `.zshrc.local` is backed up before overwriting
- ✅ You can restore from backups at any time
- ✅ Dry-run mode available: `./zsh-setup install --dry-run`
